# Claude Design Importer

You are porting a static design **mockup** into production components in an
existing codebase. Mockups — especially those generated by Claude's design tool
or similar — are **pixel-exact visual artifacts**: inline `style="..."`, hardcoded
`font-size:13.5px`, hardcoded px padding, raw `<div>`s, literal hex colors. They
are made to *look* right in isolation, not to be a component system.

A literal 1:1 translation is the failure mode this skill exists to prevent. Copying
`font-size:13.5px` → `text-[13.5px]`, `padding:9px 11px` → arbitrary px padding, and
`color:#7C5CFC` → `text-[#7C5CFC]` produces code that ignores the type scale, the
spacing scale, and the user's root font size. It looks identical and is unmaintainable
and inaccessible.

## The golden rule

**The mockup defines the intended look, not the implementation.** Match its visual
result, but re-express everything through the project's token system and component
primitives. Treat every literal value in the file — px sizes, hex colors, px spacing —
as an *approximation to snap onto the nearest scale token*, never as a value to copy.
Half-pixel sizes (`13.5px`, `11.5px`) are noise from a design tool, not intent.

## Step 1 — Preflight: inspect, then output a PLAN and STOP for approval

Do not write any component code yet.

1. **Detect the stack.** Inspect the repo: Tailwind v4 (tokens in CSS via `@theme`)
   vs v3 (`tailwind.config`); whether shadcn/ui + Radix are present (`components/ui`);
   the existing color tokens, fonts, and any current type/spacing scale. Adapt all
   output to what's actually there — reuse it, don't reinvent it.

2. **Inventory the mockup.** List every distinct font-size, line-height,
   letter-spacing, color, radius, and spacing value used.

3. **Propose a type scale.**
   - Base/body = **16px (1rem)** unless you can justify otherwise; never inherit a
     small mockup base (mockups often sit at 13–14px, which is why they read tiny).
   - Derive the ramp from a **single modular ratio**: Major Third **1.25** for dense
     tool UI, Perfect Fourth **1.333** for more heading contrast.
   - Express every step in **rem**, so it honors the user's root font size and zoom.
   - **Each step bundles its own tuned font-size + line-height + letter-spacing.**
     Line-height multiple *tightens* as size grows; large headings take slightly
     *negative* tracking. Do not reuse one line-height ratio across the scale — that
     is the precise technical reason a uniform multiply is wrong.
   - Produce a **mapping table**: each mockup px size → nearest scale step
     (e.g. `13.5px → text-sm (0.875rem)`, `19px → text-xl`).

4. **Map colors to existing semantic tokens.** Mockup vars/hex (`--ink`, `--accent`,
   `#7C5CFC`) → the project's existing tokens (`text-foreground`,
   `text-muted-foreground`, `bg-card`, `border`, `primary`…). Only introduce a new
   token when a color has no home — and flag it for a human.

5. **Map spacing to the spacing scale.** Mockup px padding/gaps → the project's
   scale (8px family: `p-2`, `gap-3`, …). Flag any value that doesn't fit a step.

6. **Map elements to components.** Which mockup divs become which shadcn/Radix
   primitives (`Button`, `Card`, `Dialog`, `Badge`, `Avatar`, `Input`, `Tabs`…)
   instead of bespoke styled divs.

Present this plan and wait for approval before generating code.

## Step 2 — After approval, generate components under these hard rules

- **No arbitrary type values.** Never emit `text-[Npx]` or `leading-[Npx]`. Use named
  scale tokens (`text-sm`/`base`/`lg`…) or `@theme`-defined `--text-*` in rem.
- **rem for all type and scalable sizing.** Reserve px for true hairlines (1px borders).
- **Reuse, don't restyle.** Use existing shadcn components and semantic color tokens;
  don't rebuild a button from a styled div or hardcode a hex that has a token.
- **Spacing and radius come from the scale**, not arbitrary px.
- **Accessibility floor:** semantic HTML (real headings, landmarks, labels), visible
  `focus-visible` states, correct ARIA where Radix expects it, respects reduced motion.
- **No orphan magic numbers.** Every value must trace to a token. If something genuinely
  cannot map, STOP and ask — do not invent a number.

## Step 3 — Self-check before finishing

Grep your own output for `[…px]`, raw hex colors (`#`), and inline `style=`. Report
every hit and either justify it (true hairline, one-off gradient stop) or fix it.

## Why this works

Professional design systems (Radix Themes, IBM Carbon, Material 3, Atlassian) never
scale a UI by multiplying every value by a constant. They set a correct base and let a
token system carry it: rem-based type tokens with per-step tuned line-height and
tracking, a separate 8px-based spacing scale, and — where users need bigger/denser —
a discrete density/scale setting rather than a free-form multiplier. This skill makes
the importer follow that discipline instead of freezing a mockup's magic pixels into
the codebase.
