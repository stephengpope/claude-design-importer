# Claude Design Importer

You are porting a static design **mockup** into production components in an
existing codebase. Mockups — especially those generated by Claude's design tool —
are **pixel-exact visual artifacts**: inline `style="..."`, hardcoded
`font-size:13.5px`, hardcoded px padding, raw `<div>`s, literal hex colors. They
are made to *look* right in isolation, not to be a component system.

A literal 1:1 translation is the failure mode this skill exists to prevent. Copying
`font-size:13.5px` → `text-[13.5px]`, `padding:9px 11px` → arbitrary px padding, and
`color:#7C5CFC` → `text-[#7C5CFC]` produces code that ignores the type scale, the
spacing scale, and the user's root font size. It looks identical and is unmaintainable.

## The golden rule

**The mockup defines the intended look, not the implementation.** Match its visual
result, but re-express everything through the project's token system. Treat every
literal value — px sizes, hex colors, px spacing — as an *approximation to snap onto
the nearest existing token*, never as a value to copy. Half-pixel sizes (`13.5px`,
`11.5px`) are noise from a design tool, not intent.

## This is a SNAP-TO-EXISTING job, not a design job

Do **not** invent a new modular type scale or hand-tune line-heights. That is
greenfield design work. Importing is mechanical: **map each value to the nearest token
the framework/project already provides** (Tailwind's default `text-*`, the spacing
scale, the existing color/radius tokens). The defaults already ship tuned
line-heights. Snapping is deterministic — there is nothing to "approve" up front.

Nearest-token rule (tie → round to the larger, for legibility): e.g. with Tailwind
defaults, `13.5px → text-sm (14)`, `11px → text-xs (12)`, `27px → text-3xl (30)`.
Spacing on the grid (`px % 4 == 0`) converts to a pixel-identical token (`w-[272px] →
w-68`); off-grid optical values snap to the nearest step.

## Decide deterministically; do NOT hand the user mechanical choices

The mapping is yours to compute and apply. Do not stop to ask "text-sm or text-base
for body?" — snap to nearest and move on. Escalate ONLY genuine product decisions
(e.g. "this color has no token at all — add `--color-success` or leave it?"). Burning
the user's time on choices the nearest-token rule already answers is the main way this
task goes bad.

## Sweep EVERY category — do not declare victory early

Arbitrary values hide in more places than font size. Convert all of them:

- **Typography** — `text-[Npx]` and `text-[Nrem]` → named scale.
- **Spacing & dimensions** — `p-/m-/gap-/w-/h-/size-[Npx]` → spacing scale.
- **Colors** — `bg-/text-/border-[#hex|rgba()]` → existing semantic tokens; if a color
  genuinely has none, add one token (don't hardcode).
- **Radii** — `rounded-[Npx]` → the radius scale.
- **Shadows** — `shadow-[…]` → shadow tokens.
- **Letter-spacing** — `tracking-[…]` → tracking tokens.

After each category, grep the WHOLE codebase to confirm that category is at zero.

## Know exactly where the line is

Three buckets — treat them differently:

1. **Violations → tokenize** (everything above). Systematic arbitrary values.
2. **Legitimately arbitrary → leave** (this is correct Tailwind, not a dodge):
   viewport units (`max-h-[86vh]`), `calc()`, hairline `border-[1.5px]`, custom grid
   templates (`grid-cols-[1fr_148px_44px]`), one-off transforms. No token exists for
   these.
3. **Already correct → never touch:** variant selectors like `data-[state=open]`,
   `has-[…]`, `group-[…]`, `aria-[…]`. These are not values — "fixing" them breaks code.

**Vendored primitives (shadcn `ui/*`) ARE in scope.** They're your owned source; the
CLI copied them in to be edited. Tokenize their arbitrary values too. The ONE real
caution: **never blindly redefine a shared scale token** (e.g. `--radius`, base
spacing) — that cascades to every component. Map onto the existing scale; don't
redefine the scale itself unless that's the explicit goal.

## VERIFY VISUALLY — this is not optional, and grep is not enough

A text-level "no `[px]` remain" check does not prove the UI still looks right. Render
it and compare:

1. **Detect the stack and run the app** (or use the running dev server).
2. **Screenshot affected surfaces before, apply, screenshot after, and compare.**
3. **Cover the states that hide regressions** — not just the default view:
   - **Auth-gated routes** (log in via the API to get a session cookie; drive headless
     Chrome / puppeteer-core with it — don't give up because a page needs auth).
   - **Dark mode.**
   - **Populated** vs empty (a real list/board/conversation, not just the empty state).
   - **Sub-tabs, modals, dropdowns** — where most edits often live.
4. **Empirically confirm the framework GENERATES the classes you emit** — compile the
   CSS and grep for the rule (e.g. confirm `.w-195` exists and equals the right px).
   Do not assume an unusual utility resolves. Watch for false negatives (CSS escapes
   dots: `.w-0\.5`).
5. **Run the production build and the linter** before claiming done.

## Report honestly

- **Read the full `git diff` before describing what you changed.** Do not claim scope
  ("I left X untouched") you haven't verified against the diff.
- State what's verified vs not. If a surface couldn't be rendered (no data exists,
  etc.), say so plainly instead of implying it was checked.
- "Done" means: every category at zero, build passes, lint clean, tokens confirmed
  generating, and the affected surfaces screenshotted in light + dark.

## Why this works

Professional design systems never freeze a mockup's magic pixels into the codebase.
They snap values onto a constrained, named scale (rem-based type with tuned
line-heights, an 8px spacing scale, semantic colors), keep the framework's existing
tokens, and verify the rendered result. This skill makes the import follow that
discipline — and, critically, makes you *look at what you built* before calling it
done.
